1. Endpoint Security Engineering

Reading time: 75 minutes | Deep dive into device hardening, defense-in-depth, and secure lifecycle

Foundational Concepts: The Endpoint as Attack Surface

In real incidents, endpoints are where attacker intent becomes visible: suspicious child processes, abnormal auth prompts, token theft activity, and outbound beaconing. Endpoint security engineering is not just "hardening checklists". It is building a system that lets analysts answer quickly: what ran, who ran it, where it spread, and whether containment worked.

  • Prevent attacks from executing (hardening, policy enforcement)
  • Detect when attacks succeed despite prevention (telemetry, anomaly detection)
  • Respond rapidly to minimize blast radius (isolation, containment, remediation)

The practical objective is controlled failure: assume one layer will fail, then design the next layer to detect and limit blast radius with evidence you can act on in minutes, not hours.

The Six Layers of Endpoint Security

Hardening Baselines: From Policy to Enforcement

Industry standards provide hardening guidance, but implementation requires translating policy into enforceable configuration management:

CIS Benchmarks (Center for Internet Security)
CIS provides detailed checklists for Windows, macOS, and Linux with two levels: Level 1 (Foundational—essential controls suitable for most environments) and Level 2 (Advanced configuration with operational overhead). A mature security program implements Level 1 across all endpoints and Level 2 for high-value systems (executives, data center access, payment processing).

Secure Device Lifecycle

Security must be managed across the entire device lifespan, not just active use:

1. Procurement - Verify hardware authenticity and security certificates; validate BIOS/UEFI integrity; confirm TPM 2.0 presence and functionality
2. Provisioning - Deploy hardened OS image; configure baselines automatically; enroll in MDM/Intune; apply security group policies
3. Active Use - Continuous patch management, compliance monitoring, threat detection, and incident response
4. Refresh Cycles - Reapply baselines when OS versions update; validate compliance; test new security features
5. Decommission - Secure data wipe per NIST 800-88 guidelines; destroy encryption keys; dispose following environmental/data protection regulations

Patch Management: Balancing Speed and Risk

Unpatched systems are the #1 exploitable vulnerability in most enterprises. However, reckless patching causes business disruption. Modern patch strategy uses SLA tiers and measured rollout:

Critical 9.0-10.0 48 hours High-risk pilot (10%) → 25% → 100% Rescan, endpoint health, EDR signal
High 7.0-8.9 7 days Pilot → Stage weekly App testing, user acceptance testing
Medium 4.0-6.9 30 days Monthly deployment window Functional testing

Patch SLA Enforcement: Track compliance daily. A single critical vulnerability 72 hours unpatched represents material risk to the business. Build dashboards showing remediation velocity by department and exceptions with business justification. Escalate SLA violations to leadership.

EDR Deployment & Tuning

Endpoint Detection and Response (EDR) agents provide continuous telemetry on process execution, file operations, network connections, and user activity. deploy in phases:

  • Pilot Phase (1-2 weeks): Deploy agents to 5-10% of organization; confirm EDR can reach all systems; measure CPU/memory impact
  • Baseline Phase (2-4 weeks): Collect clean network telemetry to establish "normal" behavior; tune detection rules to reduce false positives
  • Alert Phase (1-2 weeks): Enable alerting; verify alert routing, triage process, and response procedures work end-to-end
  • Full Rollout: Deploy to entire organization with phased, monitored approach

Endpoint Baseline vs Advanced Controls

  • Full-disk encryption active
  • Antivirus/EDR deployed and reporting
  • OS patched within SLA
  • Screen lock at 5 min
  • Local admin removed
  • Firewall enabled
  • USB storage restricted
Level 2: Advanced (Priveleged/Sensitive)
  • Application allowlist enforced
  • Exploit protection (EMET/Exploit Guard)
  • Credential Guard active
  • Controlled Folder Access
  • Network isolation on anomaly
  • Hardware security key for login
  • Memory integrity (HVCI)
Level 3: Executive / Sensitive (Highest Risk)
  • Managed device only (no BYOD)
  • Zero-trust network access only
  • No local data storage permitted
  • Enhanced logging forwarded in real-time
  • Dedicated threat hunting coverage
  • Physical security requirements
  • Periodic forensic hygiene checks

Hands-On Tutorial

Endpoint Triage with osquery: Investigating a Live Host

Scenario: Your SIEM fired an alert: a PowerShell process on WKSTN-0231 connected outbound to an unknown IP. You have osquery on the endpoint and 20 minutes before the incident commander wants a scope assessment.

1
Connect to the endpoint via osquery remote or live response shell.

If using CrowdStrike RTR: runscript -CloudFile="osquery_triage.sh" -CommandLine="--hostname=WKSTN-0231". For Defender Live Response: upload and run the script. For direct access: osqueryi --json.

2
Find what is listening on the network right now.
-- Query: All processes with active network connections SELECT p.pid, p.name, p.path, p.cmdline, l.port, l.protocol, l.address FROM processes p JOIN listening_ports l ON p.pid = l.pid WHERE l.address NOT IN ('127.0.0.1','::1') ORDER BY l.port;

Flag any process you do not recognize on any port. Python on 8443 or unknown binaries from /tmp are immediate escalation signals.

3
Check for persistence: startup items and scheduled tasks.
-- Startup persistence across all known vectors SELECT name, path, source, status FROM startup_items ORDER BY source; -- Scheduled tasks (Windows-equivalent via crontab table on macOS/Linux) SELECT command, minute, hour, path FROM crontab WHERE command NOT LIKE '%/usr/bin%'; -- LaunchAgents and LaunchDaemons (macOS) SELECT name, program, program_arguments, run_at_load FROM launchd WHERE run_at_load = '1' AND NOT name LIKE 'com.apple.%';

Any entry with a path in /tmp, /var/folders, or a user home directory that you cannot attribute to a known application is suspicious. Hash the binary immediately.

4
Get the hash of any suspicious file for threat intel lookup.
-- Hash suspicious files found in startup scan SELECT path, md5, sha256 FROM hash WHERE path IN ('/tmp/.hidden/updater_v2', '/Users/jsmith/.local/bin/svc_helper'); -- Cross-reference: file was modified recently (within 24h)? SELECT path, mtime, ctime, size FROM file WHERE path = '/tmp/.hidden/updater_v2' AND mtime > (strftime('%s','now') - 86400);

Take the SHA256 and check VirusTotal, MalwareBazaar, or your SIEM threat intel feed. A detection rate of any positive hit = escalate to SEV2 immediately.

5
Check user accounts for privilege escalation signs.
-- Users with interactive shells (flag service accounts with shell access) SELECT uid, gid, username, shell, directory FROM users WHERE shell NOT IN ('/usr/bin/false','/usr/sbin/nologin','/bin/nologin'); -- Recent login history SELECT username, tty, host, time, pid FROM last ORDER BY time DESC LIMIT 20; -- Sudo usage in the last hour (Linux systems) SELECT username, rule_details, last_used FROM sudoers WHERE last_used > (strftime('%s','now') - 3600);

Look for service accounts that have a login shell (svc_deploy with /bin/bash is a risk), unknown users created recently, or sudo escalation for accounts with no operational reason.

6
Package your findings into the IR ticket.

Required artifacts: (a) osquery output export in JSON, (b) SHA256 of any suspicious file, (c) process tree screenshot, (d) network connection log timestamp, (e) list of all persistence entries found. Decision point: if you found an unknown persistent binary or an active outbound connection to a non-corporate IP, escalate to containment. Isolate the host before remediation — never remediate evidence in place without approval.

2. Network Security & Zero Trust Architecture

Reading time: 80 minutes | Zero trust model, microsegmentation, and continuous verification

The Evolution from Perimeter-Based to Zero Trust

Traditional Model (Perimeter Security): Trust everything inside the network boundary, block everything outside. This assumed a clear network perimeter, true before cloud computing and remote work became ubiquitous.

Zero Trust Model: Never trust, always verify—regardless of network location. Every access request requires validation of identity (who?), device posture (is it healthy?), context (where? when?), and least-privilege scope (what?). Zero Trust is not a product; it's an architecture philosophy requiring rethinking of how access decisions are made.

Zero Trust Pillars: Identity, Device, Application

Microsegmentation: From Flat Networks to Trust Zones

Microsegmentation divides the network into small, isolated trust zones, reducing blast radius when breaches occur and limiting lateral movement. A modern network uses application-centric segmentation rather than IP-based segmentation.

Segmentation Strategy by Business Function

Zone A: Development & QA - Developers, QA, CI/CD pipelines - Assume dependencies from external sources (untrusted) - Allowed outbound: Any (with logging) - Allowed inbound: Monitored, logged for incident investigation - Threat level: High (experimenting with code) Zone B: Production - Production applications, databases, customer-facing services - Allowed inbound: Only from load balancers (HTTPS/TLS encryption) - Allowed outbound: Only to approved services (AWS SQS, Azure Storage, etc.) - Denial of Service: Immediate blocking, incident investigation - Threat level: Critical (protect customer data) Zone C: Security Infrastructure - SIEM, EDR management servers, vulnerability scanners - Isolated zone: minimal inbound (sensor collection only) - Sensitive: Access logged and audited heavily - Threat level: Critical (if compromised, all defenses fail) Zone D: Guest / Contractors - Limited network access (WiFi isolation) - Cannot reach internal resources - Egress monitored heavily for suspicious patterns - Threat level: Medium (external parties) Firewall Baseline Rules: DENY all by default Each allowed flow must have documented business justification and owner Review quarterly for drift and effectiveness
Segmentation Implementation Caution:
Too aggressive segmentation breaks legitimate business processes and creates support ticket burden. Start with critical data flows (production → database, user → production). Test thoroughly. Expand in phases. Every rule should have documented business justification and owner. Measure by MTTR (mean time to remediate) for incidents caught by segmentation.

Network Monitoring & Threat Detection

Even with strong segmentation, continuous network monitoring detects suspicious behavior:

  • Beaconing: Regular outbound connections to known C2 infrastructure (indicator of compromise/persistence)
  • Port Scanning: Reconnaissance activity indicating lateral movement attempts
  • Data Exfiltration: Unusually large traffic volumes to external IPs (data theft)
  • DNS Tunneling: Data hidden in DNS queries to evade detection (command & control)
  • Protocol Anomalies: Unusual port combinations, traffic patterns, encryption usage (evasion technique)
  • Lateral Movement: Unexpected east-west traffic between zones normally isolated

Conditional Access & Risk-Based Authentication

Conditional Access engines (Azure AD, Okta) evaluate context in real-time to adjust required authentication strength:

  • Low Risk Context: Corporate network + registered device + known user IP = Standard MFA
  • Medium Risk Context: Personal WiFi + unregistered device = Require number matching MFA or security key
  • High Risk Context: Unknown IP + unusual location + multiple failed attempts = Block and escalate for investigation

Key Learning Outcomes

  • Understand zero trust architecture principles and deployment fundamentals
  • Design multi-layer identity verification: MFA, risk scoring, continuous authentication
  • Implement device health checks with real-time compliance scoring and enforcement
  • Plan network microsegmentation by trust zone and business function
  • Monitor networks for lateral movement, beaconing, and data exfiltration indicators
  • Configure conditional access policies balancing security and usability

  • Small, consistent payload sizes
  • Connection to newly-registered or low-reputation domains
  • HTTPS to non-standard ports (443 on unusual IPs)
Lateral Movement Signals
  • New SMB/RDP connections between workstations
  • Admin share access (C$, ADMIN$) outside change windows
  • WMI remote execution from unusual source
  • Pass-the-hash/Pass-the-ticket Kerberos anomalies
Data Exfiltration Indicators
  • Unusual volume spikes in outbound traffic
  • DNS queries with long subdomains (tunneling)
  • Upload volume to new cloud storage endpoints
  • Compressed archive creation followed by transfer
C2 Infrastructure Signals
  • Low TTL DNS responses with frequent IP changes
  • Domain generation algorithm (DGA) patterns: random-looking domains
  • HTTPS with self-signed or mismatched certificates
  • Traffic to TOR exit nodes or known proxy services

3. Penetration Testing & Assessment Methodology

Reading time: 85 minutes | Assessment frameworks, exploit validation, risk communication

Purpose & Scope of Penetration Testing

Penetration testing should mirror how attackers chain weaknesses in production: initial access, privilege expansion, lateral movement, and objective completion. The value is not the screenshot of a vulnerable service. The value is proving whether your preventive and detective controls hold under realistic attack paths.

Critical Principle: Report control failure with business consequence, not just vulnerability presence. If exploitation is blocked by compensating controls and validated evidence supports that, document it as control effectiveness, not noise.

Assessment Methodologies

  • NIST SP 800-115: Federal standard for technical security testing; provides detailed methods for vulnerability discovery and exploitation
  • OWASP Testing Guide (OTG): Web application security testing methodology with 100+ detailed test cases organized by vulnerability type
  • PTES (Penetration Testing Execution Standard): Industry consensus on phases, from scoping through post-engagement activities
  • MITRE ATT&CK: Framework for mapping adversary tactics and techniques observed in real attacks; use to scope testing to realistic threat scenarios

Pentesting Lifecycle

Hands-On Tutorial

Web Application Pentest Chain: Recon → Exploitation → Reporting

Scenario: You have a scoped assessment against a staging e-commerce application at 10.10.10.45. Scope: web application only (no social engineering, no network pivot). Time box: 8 hours. Deliverable: written findings report with CVSS scores and proof-of-concept screenshots.

1
Passive and active reconnaissance — build the attack surface map.
# DNS enumeration — find all subdomains in scope subfinder -d corp-staging.internal -o recon/subdomains.txt dnsx -l recon/subdomains.txt -a -cname -o recon/dns-resolved.txt # Port and service fingerprinting nmap -sV -sC -p 21,22,80,443,3306,3389,8080,8443 --open 10.10.10.45 -oN recon/nmap.txt # Web technology fingerprinting whatweb http://10.10.10.45 --aggression 3 | tee recon/whatweb.txt wappalyzer http://10.10.10.45 # identifies CMS, frameworks, libraries

Key output: open ports, software versions, server OS, CMS/framework. Apache 2.4.29 means check for CVE-2021-41773 (path traversal RCE). MySQL 5.7 on 3306 means check if internet-accessible — that alone is a High finding.

2
Directory and endpoint discovery — find hidden attack surface.

4. Security Architecture Design & Risk Modeling

Reading time: 75 minutes | ADRs, threat modeling, control mapping

Architecture as Security Control

Architecture decisions determine whether incidents become isolated events or enterprise outages. In practice, most expensive security failures are design failures: over-trusted networks, weak identity boundaries, and missing ownership of critical data paths. This section focuses on decisions that survive real operational pressure.

Architecture Decision Records (ADRs)

Capture the "why" because teams rotate, systems evolve, and incident response depends on historical decisions. A useful ADR explains constraints, rejected alternatives, and rollback criteria so responders can reason quickly when assumptions fail.

# ADR-001: Mandatory Full-Disk Encryption for All Endpoints Status: Accepted Date: 2026-01-15 Stakeholders: CISO, IT Operations, Legal, CFO ### Context - Regulatory requirement (HIPAA, GDPR): Encrypted storage of sensitive data - Increasing theft/loss of company devices (5+ incidents this year) - Performance concern about encryption overhead on older laptops - Support cost of password resets and recovery ### Decision Implement BitLocker (Windows) / FileVault (macOS) with mandatory encryption on all endpoints by 2026-02-15, with opt-out period for existing devices showing hardship. ### Rationale - Regulatory compliance (mandatory for covered systems under HIPAA/GDPR) - Mitigates risk if devices lost/stolen (impossible to access without encryption key) - Modern hardware mitigates performance impact (~2-5% on SSDs, 10-15% on older spinners) - Escrow recovery keys to IT helpdesk reduces support burden vs. password-only recovery ### Alternatives Considered 1. **Optional encryption** (rejected: only 15-20% adoption rate in test, leaves risk) 2. **Hardware-only encryption** (rejected: not all devices support, doesn't protect all data) 3. **Defer 6 months** (rejected: regulatory deadline is non-negotiable) 4. **Full-disk encryption + passphrase escrow** (accepted - provides both security and recovery) ### Trade-offs - Performance impact: ~2-5% on SSDs, ~10-15% on spinning disks - Recovery key management overhead (escrow to IT) - Support calls for password resets (mitigation: strong self-service reset) - Initial deployment: 2-3 months to roll out to 5,000+ devices ### Implementation Plan - **Phase 1 (Week 1-2):** Pilot with IT department + security team (100 devices) - Measure: Performance baseline, support ticket volume, user satisfaction - Success criteria: >95% devices encrypt successfully, MTTD (mean time to deploy) < 15 min/device - **Phase 2 (Week 3-6):** Staged rollout to executives + data handlers (500 devices) - Monitor: Boot times, application performance, issue patterns - **Phase 3 (Week 7-12):** Wave deployment to all remaining devices (4,400 devices) - Track: Adoption compliance, exception requests, support costs vs. baseline ### Success Metrics - Encryption adoption: 95%+ within 60 days of rollout deadline - User impact: < 0.5% of users unable to access systems due to encryption - Data protection: Zero data breaches attributed to unencrypted endpoint storage - Support cost: < 10% increase in IT helpdesk volume - Business continuity: Production uptime > 99.9% (no outages from encryption) ### Rollback Plan If Phase 1 shows >5% failure rate or >30% performance degradation: - Pause deployment; revert pilot devices to non-encrypted state - Adjust implementation approach (different tool, staggered timeline) - Require approval from CISO + CFO before resuming

Threat Modeling & Attack Trees

Threat modeling is most useful when it changes implementation before release. Use STRIDE to force complete coverage, then prioritize by operational impact: what could break revenue, expose regulated data, or block response capability if abused.

Threat: Attacker enrolls rogue device into MDM using stolen corporate credentials

Mitigation 2: Require admin approval gate for new device enrollments

Trust Boundaries & Data Flow Diagrams

Map trust boundaries like incident responders do: where identity context changes, where data crosses ownership domains, and where logging must be authoritative during investigations.

  • Boundary Between User Device and Corporate Network: All traffic must be authenticated, encrypted, monitored
  • Boundary Between Application and Database: Application authenticate to database, encrypt data in transit, log all queries
  • Boundary Between Corporate Network and Internet: All outbound connections filtered; inbound blocked by default

Threat Modeling Process Flow

1
Define Scope & System — Draw data flow diagram showing all actors, systems, data stores, and communication channels. Include all trust boundaries.
2
Identify Threats (STRIDE) — For each component and trust boundary, enumerate threats in each STRIDE category. Use threat library templates to speed this up.
3
Score Risk — Rate each threat by likelihood and impact (DREAD, CVSS, or qualitative). Prioritize high-likelihood + high-impact threats first.
4
Select Mitigations — For each prioritized threat, select one or more controls: prevent (design), detect (monitor), or accept (document risk with owner sign-off).
5
Validate Controls — Test whether mitigations actually prevent or detect each threat. Document evidence of effectiveness.
6
Track & Update — Review threat model when system changes. Any new feature, integration, or data type should trigger a re-review of affected boundaries.

Shifting Security Left in Software Development

Traditional Model: Teams discover critical issues late in release cycles, creating emergency fixes, deployment delays, and rollback risk.

DevSecOps Model: Security decisions are enforced where code and infrastructure change: pre-commit checks, PR gates, build-time scanning, and post-deploy drift detection. The goal is predictable risk control with minimal delivery friction.

Infrastructure-as-Code (IaC) Security

IaC security maturity is measured by change safety, not tool count. Every high-risk config change should have review evidence, expected blast radius, rollback path, and post-change validation. Without that, automation just makes bad changes happen faster.

IaC Security Best Practices:
✓ Never hardcode secrets in configs (use secret management: Vault, AWS Secrets Manager)
✓ Enforce encryption by default on all storage resources
✓ Require least-privilege IAM roles (specific actions/resources, no wildcards)
✓ Implement tagging standards for cost allocation and security grouping
✓ Scan all IaC before deployment with Checkov, Terraform Sentinel, OPA/Rego
✓ Version all infrastructure changes; require approval before deployment
✓ Maintain drift detection: compare current state vs. desired state

7. SIEM & Detection Engineering

Reading time: 80 minutes | Detection rules, ATT&CK mapping, tuning for precision

SIEM is a Practice, Not a Product

SIEM performance is an operational outcome: can analysts identify true threats quickly with low rework? Detection quality depends on hypothesis clarity, telemetry reliability, and disciplined tuning governance with explicit ownership.

Detection Logic Design: Hypothesis-Driven

Frame each detection as a falsifiable hypothesis: what attacker behavior should exist, what fields should prove it, and what benign patterns should be excluded without hiding real risk.

Example: Detect Credential Dumping (ATT&CK T1003)
Hypothesis: An attacker attempts to dump credentials from LSASS process memory
Signal: lsass.exe accessed by non-Windows DLL process + large data read (>1MB)
Rule Logic: (process_name != "svchost.exe") AND (target_process="lsass.exe") AND (memory_read > 1024000)
Risk Score: True positive = 8.5, False positive = -2.0
Tuning: Exclude known safe processes (debuggers, monitoring tools) to reduce false positives

MITRE ATT&CK Mapping

Map detections to ATT&CK so coverage gaps are visible and testable. This prevents rule sprawl by linking detections to specific attacker goals and expected analyst actions.

  • **Reconnaissance:** Network scanning, OSINT gathering, account discovery
  • **Initial Access:** Phishing, exploit kits, default credentials
  • **Execution:** Script execution, command-line usage, scheduled tasks
  • **Persistence:** Registry persistence, scheduled tasks, service installation
  • **Privilege Escalation:** UAC bypass, kernel exploitation, privilege abuse
  • **Defense Evasion:** Obfuscation, disabling security tools, process injection
  • **Credential Access:** Credential dumping, brute force, keylogging
  • **Discovery:** System network configuration, process discovery, lateral movement reconnaissance
  • **Lateral Movement:** RDP, SMB, Kerberos delegation abuse
  • **Collection:** Data from clipboard, screen capture, email collection
  • **Exfiltration:** DNS tunneling, long-connection uploads, C2 communication
  • **Impact:** Ransom, deletion, resource denial

Alert Tuning While minimizing False Positives

Detection Rule: Suspicious PowerShell Script Execution # Initial Rule (High False Positives) process.name:powershell AND command_line:*-nop* OR command_line:*-enc* # Tuned Rule (Precision-focused) process.name:powershell AND ( (command_line:*-nop* OR command_line:*exec* OR command_line:-bypass*) AND NOT parent_process:*\windows\system32\* AND NOT user:SYSTEM AND NOT user:NT_AUTHORITY ) AND NOT ( device.hostname:dev-* OR device.hostname:test-* ) AND risk_score >= 7.0 # Result: Reduced false positives from 50/day to 2/day # Sensitivity: Still catches 95% of real threats (measured in controlled testing)

8. Security Operations & Incident Response

Incident Response as a Discipline

Incident response quality comes from repeatable execution under uncertainty: accurate severity calls, fast containment, clean handoffs, and evidence-rich communication to technical and business stakeholders.

The IR Lifecycle

Phase 1: DETECT & Alert - Alert triggered in SIEM or monitoring system - Alert routing rules determine who reviews (L1 Analyst, SOC Team Lead, etc.) - Alert enrichment: Add context (device owner, criticality, recent changes) Phase 2: TRIAGE - Analyst reviews alert + context - Classify severity: Critical / High / Medium / Low - Determine response: Immediate action / Investigation / Monitoring - Escalate if needed: CISO, legal, incident commander Phase 3: CONTAIN BLAST_RADIUS - CRITICAL incidents: Isolate affected devices from network immediately - Revoke tokens/sessions for compromised users - Block suspicious IPs/domains at firewall - Preserve evidence: Capture memory dumps, disk images (before containment) Phase 4: INVESTIGATE - Forensics team analyzes preserved evidence - Timeline reconstruction: What happened, when, how did attacker enter? - Scope assessment: How many accounts/devices compromised? - Root cause identification: What control failed? Phase 5: REMEDIATE & RECOVER - Patch vulnerable system or rebuild from clean snapshot - Reset compromised credentials - Revoke attacker access (SSH keys, tokens, etc.) - Validate fix effectiveness: Retest to confirm remediation Phase 6: COMMUNICATE & DOCUMENT - Incident report for stakeholders (technical + executive summary) - Regulatory notifications if required (GDPR takes 72h, others vary by law) - Customer notification if data breach confirmed - Lessons learned meeting with team Phase 7: LESSONS LEARNED & IMPROVEMENT - Root cause analysis: Why did controls fail? - Permanently fix root causes (not just symptoms) - Update detection rules based on observed IOCs - Conduct tabletop exercises with team to practice for similar incidents

9. Threat Modeling & Risk Analysis

Reading time: 75 minutes | STRIDE framework, attack trees, control validation

Why Threat Modeling Matters

Threat modeling is a release safety control. It forces teams to surface abuse paths before code reaches production, when mitigations are cheaper and less disruptive than post-incident redesign.

STRIDE Threat Categories

  • Spoofing: Attacker impersonates someone/something (fake identity, forged token, credential theft)
  • Tampering: Attacker modifies data in transit or at rest (MITM, SQL injection, file modification)
  • Repudiation: Attacker denies their actions ("I didn't do it")—logging and audit trails prevent
  • Information Disclosure: Attacker accesses sensitive data (eavesdropping, SQL injection, weak encryption)
  • Denial of Service: Attacker makes system unavailable (network saturation, resource exhaustion, crashes)
  • Elevation of Privilege: Attacker gains higher permissions than authorized (privesc exploits, misconfigured RBAC)

Threat Model Example: API Authentication Flow

System: Mobile App ← HTTPS → API Server ← Database STRIDE Analysis: SPOOFING (who am I?) Threat: App makes request with forged JWT token Mitigation: Validate JWT signature, check expiration, require HTTPS only Threat: Attacker spoofs API server (MITM attack) Mitigation: Certificate pinning in mobile app, HTTPS with TLS 1.3+ TAMPERING (integrity) Threat: Attacker modifies request in transit (MITM) Mitigation: HTTPS encryption, request signing with HMAC REPUDIATION (accountability) Threat: User denies they made a purchase Mitigation: Log all API calls with user ID, timestamp, request signature INFORMATION DISCLOSURE Threat: API returns too much data (e.g., user emails in list query) Mitigation: API field-level access control, return only necessary data Threat: Attacker sniffs traffic on public WiFi Mitigation: Enforce HTTPS only, certificate pinning, data encryption at rest DENIAL OF SERVICE Threat: Attacker floods API with requests Mitigation: Rate limiting, DDoS protection (AWS Shield), auth challenges ELEVATION OF PRIVILEGE Threat: Regular user modifies admin flag in JWT token Mitigation: Sign JWT server-side, never trust client-provided roles MITIGATIONS SUMMARY: 1. Enforce HTTPS/TLS 1.3+ for all traffic 2. Sign all tokens server-side; validate signatures 3. Implement rate limiting: 100 req/min per user 4. Log all API calls for audit trail 5. Field-level access control: only return authorized data 6. Certificate pinning in mobile app 7. Input validation for all API parameters

  • 7: Sensitive data exposed

R — Reproducibility
  • 10: Always reproducible
  • 7: Reproducible with custom tools
  • 5: Sometimes reproducible

  • 10: Novice using script-kiddie tools
  • 7: Skilled attacker, standard tools
  • 5: Expert attacker

10. Vulnerability Management & Patch Operations

Reading time: 80 minutes | CVSS scoring, remediation tracking, SLA enforcement

Vulnerability Management is Risk Management

Vulnerability management is triage economics: limited patch windows, competing outages, and uneven exploitability. Mature programs prioritize by exploit reality and business impact, then verify closure with evidence instead of ticket status alone.

Vulnerability Prioritization Model

Risk Score = CVSS_Base_Score × Exploitability_Factor × Exposure_Factor × Asset_Criticality CVSS_Base_Score: 0-10 (industry standard, published with each CVE) - ATT&CK (Accessible over network, low complexity, no auth): +0.5 multiplier - LOCAL (Local attack only, high complexity, auth required): +0.1 multiplier Exploitability_Factor: 1.0 = public exploit available; 0.5 = no public exploit known - Check: CISA Known Exploited Vulnerabilities (KEV) catalog Exposure_Factor: 1.0 = internet-facing; 0.5 = internal only; 0.1 = air-gapped - Check: Network scan results Asset_Criticality: 5.0 = production database; 3.0 = standard device; 1.0 = dev machine - Check: Asset management system EXAMPLES: CVSS 9.8 × 1.0 (public exploit) × 1.0 (internet-facing) × 5.0 (prod DB) = 49.0 → Patch in 24 hours CVSS 6.2 × 0.5 (no exploit) × 0.5 (internal) × 1.0 (dev device) = 1.55 → Patch in 90 days CVSS 7.5 × 1.0 × 0.5 (internal only) × 3.0 = 11.25 → Patch in 30 days

Common Crypto Mistakes to Catch in Code Reviews

  • Hardcoded keys or secrets: Any encryption key in source code is compromised at commit time. Use KMS, Vault, or Secrets Manager with rotation.
  • ECB mode cipher usage: AES-ECB leaks patterns. Require GCM or CBC with HMAC. Flag any Cipher.getInstance("AES/ECB") in Java code reviews.
  • Predictable IVs or nonces: Reusing IV with the same key in GCM mode is catastrophic. IVs must be random for every encryption operation.
  • Storing passwords with reversible encryption: Passwords must use adaptive hashing (Argon2id, bcrypt) — never AES, never SHA-256 alone.
  • Disabled certificate validation: ssl.CERT_NONE, InsecureSkipVerify: true, custom TrustManagers returning null — immediate findings.
  • Weak PRNG: Math.random() or random.random() for tokens/keys. Must use secrets module (Python) or SecureRandom (Java).

  • Communicate in impact, not just artifacts: Stakeholders need: what data was accessed, is the attacker still present, what systems are affected, what business operations are at risk. Not just raw IOCs.

  • Input validation: All user-controlled input validated server-side, not just client-side. Check for SQL injection (parameterized queries only), command injection (subprocess with list args), and path traversal (../ in file paths).
  • Authentication & session management: Passwords hashed with bcrypt/Argon2id. Session tokens are cryptographically random (≥128 bits). Sessions invalidated on logout and privilege change.
  • Authorization checks: Every endpoint checks authorization, not just authentication. Object-level authorization (IDOR checks): verify the requesting user owns the resource being accessed.
  • Secrets management: No credentials in code, environment variable dumps, or log output. Check for print(password), logging of authorization headers, or config files committed to git.
  • Error handling: Generic error messages to users; detailed errors only in server-side logs. Stack traces in HTTP responses are an immediate finding.
  • Security headers: CSP, HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy. Verify they're set in responses with curl or browser DevTools.

20. Container & Kubernetes Security

Containers and Kubernetes have fundamentally changed deployment architectures—and with them, the attack surface. Container security requires hardening images, enforcing K8s RBAC, writing network policies, scanning for vulnerabilities at build time, and detecting runtime anomalies with tools like Falco. These skills are required for cloud-native and DevSecOps roles.

            matchLabels:
              app: frontend
      ports:

Kubernetes Security Hardening — Key Points

  • Pod Security Admission: Replace deprecated PSPs with PSA labels on namespaces (enforce: restricted). Restricted profile blocks privileged pods, hostPath mounts, and non-root violations.
  • RBAC least privilege: Audit with kubectl auth can-i --list --as=system:serviceaccount:ns:sa. Remove cluster-admin bindings for non-bootstrapping components. Prefer Roles over ClusterRoles.
  • Image provenance: Pin images by digest (image: repo/app@sha256:abc123). Use OPA/Gatekeeper or Kyverno policies to enforce digest pinning and block :latest tags in production.
  • etcd encryption: Secrets are base64-encoded at rest by default — not encrypted. Enable EncryptionConfiguration for AES-GCM encryption of Secret objects in etcd.
  • Admission controllers: OPA Gatekeeper or Kyverno as policy engines. Required checks: no privileged containers, image registry allowlist, resource limits set, non-root enforcement.
